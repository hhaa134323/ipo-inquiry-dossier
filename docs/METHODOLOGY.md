# IPO 问询可比案例底稿 —— 检索 + 底稿方法论

本文档是本技能唯一的方法论事实源。所有 agent 入口文件（SKILL.md / AGENTS.md / CLAUDE.md / GEMINI.md）均指向此文件，不复制正文以免漂移。

---

## 核心原则

1. **先看问询原文，再想检索方案** —— 必须从问题原文（meta.问题原文）的全部限定出发，不得用概括短标签代替。
2. **召回不做取舍，精排逐个严判** —— 两段分离，以免早期误判漏掉潜在可比案例。
3. **一切结论落到“文件+页码”** —— 严禁编造，找不到就说找不到。
4. **原文逐字引用，不经过 AI 改写。**
5. **召回只走 grep，严禁整份缓存读入上下文** —— 召回阶段只用 grep 命中行（命中页码 + 上下文 1-2 行），绝不把整份 `.txt` 缓存读进模型；需要逐字读原文的只有精排阶段。
6. **口径词从问题原文动态生成，不绑定任何单一行业** —— 检索与精排用到的“关键口径词”必须由问题原文自身的限定词经下文扩展规则推导，半导体扩展表只是示例，严禁照搬到其他行业。（见步骤 2.2）
7. **精排粗到细级联，省 token 靠闸一** —— 精排分两道闸：闸一只读问询题目做粗筛，淘汰主题不符者；幸存者进闸二逐字精读回复打分。省 token 主要来自闸一挡掉大量无关候选，闸二对幸存者照常完整深读，不为省 token 牺牲证据完整性。（见步骤 3）
8. **机械步骤走固定脚本，判断步骤留给 AI** —— 抽取（`extract.py`）、召回 grep（`recall.py`）、按页读原文（`read_pages.py`）、渲染（`build_dossier.py`）都是确定性脚本，直接带参数调用，**严禁临场另写脚本**；口径词扩展（步骤 2.2）与 rubric 打分（步骤 3）是行业相关判断，由 AI 完成，不得写死进脚本。所有中间产物（term_map / candidates / hits / ranking_report / 渲染截图）一律落到工作目录 `_work/`（默认 `{output}/_work`），`--output` 目录只保留最终 `底稿_*.docx`。

---

## 步骤 1：准备 —— 提取 PDF 文本缓存

1. 对 `--input` 目录（默认 `./input`）下每份 PDF，调用 `scripts/extract.py`。
2. 该脚本用 pymupdf 逐页抽取文本，生成同名 `.txt` 文件，每页正文前插入一行 `[PAGE n]` 标记（n 从 1 开始）。
3. 若 `.txt` 缓存已存在则跳过，不重复抽取。
4. **绝不将整份 PDF 塞进上下文** —— 只操作缓存文件。

---

## 步骤 2：召回（高召回、机械、不取舍）

### 2.1 从问题原文生成核心检索词

以问题原文中**所有限定性成分**为起点，包括：
- 业务/产品实体（如“集成电路产品”“功率器件”“晶圆代工”）
- 财务/会计实体（如“毛利率”“折旧摊销”“期间费用率”）
- 动词/关系（如“变动趋势”“差异原因”“合理性”）
- 剔除/排除条件（如“剔除产能爬坡阶段影响”）

### 2.2 同义/口径扩展（按行业动态生成，不照搬示例表）

口径词必须**从问题原文自身的限定词推导**，不得套用任何固定行业词表（否则只在半导体一类问题上有效）。对 2.1 拆出的每个核心检索词，按以下四类机械扩展：

1. **上位词 / 下位词** —— 把实体沿“大类↔具体”两个方向展开（行业大类↔具体产品；费用大类↔具体科目）。
2. **同义 / 别称 / 英文缩写** —— 同一概念在招股书、问询回复正文里的常见替代表述。
3. **会计/监管口径变体** —— 同一指标的不同列示口径（如“率”“占比”“金额”“结构”等后缀变体）。
4. **问题特有的剔除/调整限定** —— 问题原文里出现的排除条件、调整项、特定情形（如“剔除 XX 影响”“仅 XX 阶段”），必须单独成词进入清单。这类限定往往就是精排维度 D 的判分锚点。

**口径词生成步骤（行业无关）：**
1. 逐条抄下问题原文的限定性成分（业务实体、财务实体、动作/关系、剔除/排除条件）。
2. 对每条按上面四类扩展，写成一张“原文词 → 扩展词”对照表，连同来源问题落盘到 `_work/term_map.jsonl`，供 `recall.py` 读取，使口径可回溯。
3. 由 `recall.py` 拿这张**动态表**去机械 grep，而不是任何预置的行业词表；脚本只 grep、不判断。

> **示例（仅供参考；换行业必须按上面步骤重新生成，严禁照搬此表）—— 半导体毛利率类问题的扩展表：**
>
> | 原文词 | 扩展检索词 |
> |--------|-----------|
> | 毛利率 | 综合毛利率、主营业务毛利率、毛利 |
> | 变动 | 波动、下滑、上升、趋势、差异、变化 |
> | 折旧摊销 | 折旧、摊销、固定资产折旧、无形资产摊销、折旧摊销占比、折旧率 |
> | 产能爬坡 | 产能利用率、良率提升、试生产、转固、固定资产转固、在建工程转固 |
> | 代工 | 晶圆代工、代工厂、foundry、委托加工、OEM |
> | 集成电路 | 集成电路、IC、芯片、半导体、数字IC、模拟IC |
> | 功率器件 | 功率半导体、功率器件、MOSFET、IGBT、SiC、分立器件 |
> | 研发费用 | 研发投入、研发支出、研发费用率、研发投入占比 |
> | 收入 | 营业收入、主营业务收入、销售收入、营收 |
> | 成本 | 营业成本、主营业务成本、单位成本、制造费用、原材料成本 |
>
> 换到消费、医药、城投、装备制造等行业时，上表整张作废，须按上面的生成步骤从该问题原文重新推导。

**扩展原则：** 宁可过召回也不漏。只要扩展词可能在问询回复的正文中出现，就加入搜索清单。

### 2.3 机械 grep（recall.py）

1. 调用 `scripts/recall.py`，读 `_work/term_map.jsonl` 的口径词，对 `--input` 目录下全部 `*.txt` 缓存逐词机械扫描。
2. 脚本**只记录命中页码、命中词和上下文 1-2 行**，不做判断、不做取舍。
3. 命中结果去重后合并为一个**候选清单**。
4. 候选清单写入 `_work/candidates.jsonl` 供精排阶段使用。
5. 口径词的生成是 2.2 的 AI 判断工作，不在脚本里；recall.py 只执行机械扫描。

### 2.4 候选不漏原则

- 当某一关键词命中不足 3 条时，应主动做更激进的口径扩展再搜一轮。
- 同一公司的不同轮次文件、不同版本全部纳入候选，不因“已有同一家”而排除。
- **召回的产出仅是候选清单，不做任何取舍判断。**

## 步骤 3：精排（粗到细级联，逐个候选严判）

精排分两道闸，**粗到细级联**：闸一用最便宜的信息（问询题目）淘汰主题不符的候选；只有幸存者才进闸二、付出逐字精读回复的成本。**token 红利主要来自闸一**——把“读几十页回复”挡在主题不符的候选之前。幸存者数量少，闸二对其**不裁剪深读窗口**，宁可多读也不漏证据。

```
候选清单
  |
  v  闸一·粗筛（只读问询题目，打 A/B）
  |--- 维度A=0（问询实质不同） -> 立即淘汰，不读回复，记 ranking_report
  |--- 幸存
         |
         v  闸二·精读（grep 定位 -> 逐字精读回复，打 C/D/E，不裁窗口）
         |--- 阈值未过 -> 丢弃，记 ranking_report
         |--- >=7 且无0分 -> 保留，写 hits.jsonl
```

### 3.1 闸一·粗筛（只读问询题目，省 token）

对候选清单中的**每个**候选，借召回阶段的命中页码定位，只读取其**问询题目**那一段（通常以“请发行人披露”“请说明”“请补充”开头，一般 1 页内，用 `scripts/read_pages.py --file <该公司>.txt --pages <题目页>` 逐字读取），逐字摘录对方问询原文（严禁改写），然后只打两维：

- **维度 A 同问询实质**、**维度 B 真问询先例**（rubric 见 3.3）。

判定：
- **维度 A = 0**（问的根本不是同一类问题） -> **立即淘汰**，不再读其回复正文，将该候选写入 `_work/ranking_report.jsonl` 并记淘汰原因。
- 其余候选 -> 进入闸二。

> 粗筛只读问询题目，不读动辄几十页的回复，绝大多数主题不符的候选在此被低成本淘汰。

### 3.2 闸二·精读（幸存者：grep 定位 -> 逐字精读回复）

仅对闸一幸存的候选做精读：

1. **grep 口径词定位** —— 用 2.2 生成的动态口径词在该候选的 `.txt` 缓存里 grep（可复用召回阶段已有的 candidates.jsonl），拿到命中页码作为**入口锚点**。
2. **逐字精读回复** —— 以命中页为入口，用 `scripts/read_pages.py --file <该公司>.txt --pages <起页>-<止页>` **完整读完支撑判分的回复论述**（含相邻的定量分析、表格页），不做“只读命中页”式的窗口裁剪。证据常跨页——口径/调整的传导链与其定量表格可能分散在相邻几页，宁可多读也不漏，这与“召回不漏、精排逐个严判”一致。逐字摘录答复要点（严禁改写）。
3. 在已读到的原文上补打 **维度 C / D / E**，与闸一的 A/B 合成总分。

> 幸存者数量少，对其完整深读的成本可控；真正的 token 红利来自闸一挡掉的大量无关候选，而不是在幸存者身上抠深读窗口。最终 `hits.jsonl` 里 `回复.起页/止页` 记录完整回复区间，供 build_dossier 逐字渲染（渲染由脚本完成，不进模型）。

### 3.3 Rubric 评分（每项 0-2 分）

A / B 在闸一打，C / D / E 在闸二打。

#### 维度 A：同问询实质（0-2）
| 分值 | 标准 |
|------|------|
| 2 | 问询问题与目标问题实质相同——同样在问毛利率/收入/成本等指标的变动原因、同样涉及多分类拆分、同样要求分析趋势 |
| 1 | 问询问题部分重叠——主题相近但限定条件不同（如只问综合口径不问分类口径），或仅有一半维度匹配 |
| 0 | 问询实质不同——问的是完全不同的主题（如技术路线/募投/关联交易而非该财务分析） |

#### 维度 B：真问询先例，非招股书自述（0-2）
| 分值 | 标准 |
|------|------|
| 2 | 内容来自**审核问询函的回复**，包含监管机构的明确问询（有“请说明”“请披露”等问询措辞），是真实的审核先例 |
| 1 | 内容来自问询回复，但引用的段落偏重发行人自述而非针对特定问题的回答 |
| 0 | 内容来自招股书、募集说明书等非问询文件，或完全不是审核问询语境 |

#### 维度 C：产品/行业可比（0-2）
**针对目标问题中涉及的业务/产品结构判断（该结构从问题原文提取，不预设行业）：**
| 分值 | 标准 |
|------|------|
| 2 | 候选与目标问题涉及的**全部主要业务/产品线**重合，且行业定位（商业模式、产业链环节等）可比 |
| 1 | 仅覆盖其中一部分业务/产品线，或行业定位略有差异 |
| 0 | 业务/行业完全不同，或未涉及目标问题关注的业务/产品 |

> 示例：当目标问题涉及“IC + 功率器件”双产品线时，同时覆盖两者且同为晶圆代工 = 2；只覆盖其中一条 = 1；封测/设备/材料等无关 = 0。换行业时，按该问题原文的业务结构重设此判分对象。

#### 维度 D：口径一致（0-2）
**针对问题原文中的特定剔除/调整限定判断（该限定从问题原文提取，不预设为折旧/产能爬坡）：**
| 分值 | 标准 |
|------|------|
| 2 | 明确讨论了问题所指的**特定口径/调整链条**，并有定量分析 |
| 1 | 提到该口径/调整，但未做定量分析，或分析口径不同 |
| 0 | 未涉及问题所指的特定口径/调整 |

> 示例：当问题要求“剔除产能爬坡导致的折旧摊销高于可比公司因素”时，明确讨论“产能爬坡->固定资产转固->折旧摊销增加->毛利率拉低”传导链并有定量分析 = 2。换问题时，这条链替换为该问题自己的剔除/调整限定。

#### 维度 E：可借鉴（0-2）
| 分值 | 标准 |
|------|------|
| 2 | 回复提供了可复用的分析框架、数据模型或论证逻辑，可以直接移植或类似处理 |
| 1 | 回复有参考价值，但需要大量调整才能适配目标公司情况 |
| 0 | 回复内容过于概况或特殊，无实际借鉴意义 |

### 3.4 阈值与取舍

- **总分 >= 7 分 且 各维度无 0 分** -> 保留，写入 hits.jsonl
- **总分 5-6 分，或某单项维度 0 分** -> 标注“弱相关，谨慎保留”，可作为备选但不作为主论据
- **总分 <= 4 分** -> 丢弃，不写入 hits.jsonl
- 闸一被淘汰（维度 A = 0）的候选同样记入精排报告。所有丢弃/淘汰的候选需在 `_work/ranking_report.jsonl` 记录**原因**和**缺口说明**。

### 3.5 缺口说明（写入 meta.缺口说明，必落底稿）

无论是否存在合格可比案例，只要有未覆盖的子问题或限定，都须如实输出缺口说明：哪些维度/子问题无法覆盖、最接近但未达标的案例及其差距；全部候选都不合格时更要明确写“未找到合格可比案例”。该说明写入 `hits.jsonl` 首行 meta 的 `缺口说明` 字段，由 `build_dossier.py` 在底稿「缺口说明 / 未覆盖事项」区显眼渲染——找不到就说找不到，且必须落到底稿，不允许只留在终端日志里。

## 步骤 4：输出 hits.jsonl（含评分依据）

在现有格式基础上，每个命中行新增 `判可比依据` 字段，其中“对方问询原文”和“答复要点”必须是逐字原文摘录（严禁 AI 改写），`评分` 数组逐维记录得分与依据，并带总分与是否保留。

- 第一行仍为 **meta 对象**（与原来一致，不含判可比依据）。
- meta 对象可含可选 `缺口说明` 字段（全局缺口，见步骤 3.5）；`build_dossier.py` 会据此渲染底稿「缺口说明 / 未覆盖事项」区。
- 每个保留案例的 `判可比依据.总分` 为**必填**；缺失时 `build_dossier.py` 会报错中断，而非静默输出 `?/10`。
- hits.jsonl 默认落在 `_work/hits.jsonl`（build_dossier 用 `--hits` 读取）。
- `关键锚点` 字段用于底稿中的黄色高亮渲染：每条短句 8-30 字、必须是从原文逐字截取的连续子串，问询侧与回复侧各 2-4 条。
- 精排过程全记录（含闸一淘汰与闸二丢弃的案例）输出到 `_work/ranking_report.jsonl` 备案。

---

## 步骤 5：引用纪律（硬性约束）

- 所有结论必须能落到 `文件 + 页码`。
- **严禁编造**。如果某项依据无法在缓存中找到，如实记录“未找到”。
- **严禁改写原文措辞**。引用时必须逐字一致。
- **找不到就说找不到**，不要猜测。

## 步骤 6：生成 .docx 底稿

1. 调用 `scripts/build_dossier.py`，可用 `--input` 指定 PDF/缓存目录、`--output` 指定输出目录（只放最终 docx）、`--work` 指定中间产物目录（默认 `{output}/_work`，存渲染截图等）、`--hits` 指定 hits.jsonl 路径（默认 `{work}/hits.jsonl`）。
2. 脚本读取 hits.jsonl 的首行作为 meta，在其中提取问题原文逐字打印于文首；其余行作为命中案例逐个处理。
3. 对于形式=图：用 pymupdf 将对应页渲染为 PNG（落 `_work/`）并嵌入。
4. **默认（形式=文本 / 表格）**：自动对源页做几何抽表 + 文本交错渲染，用 PyMuPDF `find_tables()` 检表、`get_text(blocks)` 取文本块，排除落在表格 bbox 内的块后按 y 坐标交错排序。
5. **表格三级兜底**：健康表（行>=2 且 列>=2 且 空单元格<50%） -> Word 真表格；坏表 -> 截图嵌入；无表 -> 正常段落。
6. **锚点高亮**：对正文段落，将每条 hit 的 `关键锚点` 短句在清理后正文上用 `str.find()` 做子串匹配，命中则拆成独立 run 加黄高亮（`WD_COLOR_INDEX.YELLOW`）；匹配不到则静默跳过，不近似改写原文。
7. 上述抽表全程为确定性几何算法，原文不过大模型，边际 token 近于 0。
8. 输出到 `--output` 目录下，文件名为 `底稿_{主题}_{日期}.docx`；**该目录只保留这份 docx**，渲染过程产生的 PNG/表格截图都落在 `_work/`。
